ΔWhat changed from v3 — and why

v3 made humans review the spec before the agent could start. v4 has them review the spec, the implementation, and the reporter approval all together at the end. Faster for the reporter; one heavier (but better-informed) review for humans.

v3 flowprevious
  1. Issue arrives
  2. Spec Generator drafts spec PR
  3. Reporter confirms intent
  4. Humans review & merge spec PR ← serial bottleneck
  5. Implementation Agent builds on a new branch
  6. Preview env up
  7. Reporter iterates
  8. DevOps reviews diff + merges
  9. Rollout
v4 flowthis revision
  1. Issue arrives
  2. Spec Generator drafts on shared branch agent/spec-<NNN>
  3. Reporter confirms intent
  4. Implementation Agent picks up immediately (no human wait)
  5. Preview env up
  6. Reporter iterates
  7. Reporter /approve
  8. CODEOWNERS review — spec + diff + conversation + agent reports
  9. Refinement loop (cap 3) — push commits or direct agent
  10. DevOps merges → rollout
↓ Saves ~24 h of reporter wait time. Humans review once, with full context.

8Top risks & mitigations

Human rubber-stamping after reporter approval

Reviewer waves through without examining diff.
↳ Security + Code agents post pre-review reports; CODEOWNERS checklist; 10% sample audit.

Wasted agent work on a flawed spec

Spec direction wrong; agent already coded.
↳ Reporter must signal /confirm or 24h pause; impl-agent self-escalates on contradictions.

Human refinement happens behind reporter's back

Reviewer changes anything without reporter visibility.
↳ Every human commit auto-pings reporter + redeploys preview; PR cannot merge until /approve or 24h silence.

Implementation loop never converges

Reporter keeps requesting changes.
↳ 5-iter reporter cap; 3-iter human cap; per-PR spend cap.

Reporter approves something broken

Bug ships even after the loop.
↳ Human review is a hard gate AFTER reporter approval.

Preview env leaks data

PII via reviewer login.
↳ Allow-list masking; one-time creds; private issues.

Per-tenant migration drift

One tenant fails, blocks wave.
↳ Per-tenant queue; failures isolated.

Tenancy boundary regression

Data leaks across tenants.
↳ Tenancy loop is issue-only; CODEOWNERS on saas_tenant_gate.

Chatbot leaks PII into a public issue

Customer name, email, payment data slips into a filed issue.
↳ 3-layer mask (deny-list → tenant allow-list → LLM cleanup); 0 leaks tracked as a hard metric; weekly random-row audit.

Chatbot files low-quality issue spam

Fuzzy reports flood GitHub.
↳ Confidence threshold ≥ 0.5 to file; clarifying-questions step before filing; per-tenant cap 5 issues/day.

Chatbot hallucinates a workaround

Bot suggests something that breaks customer's data.
↳ KB-sourced workarounds tagged kb-sourced; LLM-generated carry disclaimer; 👍/👎 telemetry surfaces bad ones.

Chatbot misclassifies a sensitive topic

Billing dispute filed as a "bug" in a public-ish issue.
↳ Sensitive-topic detector at infra/agentlab/sensitive-topics.yml owned by security-leads; quarterly classifier retraining; 5% sample audit.

Vendor outage / lock-in (v7)

Anthropic API down, GitHub Actions outage, Slack rate-limit.
↳ Hexagonal runtime; LiteLLM fallback chain (Claude → GPT-4o → Gemini); same OCI image runs on GitLab CI / K8s; notifier swap is config change. Worst-case migration ~1 week.

Adapter contract drift (v7)

New adapter silently breaks an agent.
↳ Contract test suite per port; every adapter must pass; CI blocks PRs that break the contract.

✓What "done" looks like

  • The reporter sees a real Odoo URL within ~30 min of confirming intent — no waiting on a human spec merge.
  • Humans review the package — spec, diff, reporter conversation, agent pre-review reports — once at the end with full context.
  • A single PR per issue accumulates every artifact (spec, plan, code, conversation, preview link).
  • ≥ 90% of merged PRs link to a spec or fix-brief.
  • 8 green checks gate main; 4 more checks + 24h soak gate prod.
  • A single client can roll forward or be paused independently via the wave system.
  • Six AI agents with distinct charters; humans always merge.
  • Either Railway or Fly can fail without taking customers down.
